2. 三个必须先立住的概念

2.1 tx 生命周期（一定要背下来）

①  用户点"发送" 
   ↓ (前端调 wallet 的 eth_sendTransaction)
②  钱包弹窗 → 用户签名（本地私钥签，不上链）
   ↓
③  签名后的 raw tx 被广播到节点，进 mempool
   ↓ (前端此时拿到 tx hash，但 tx 还没上链！)
④  矿工/验证者从 mempool 选 tx 打包进区块
   ↓
⑤  区块被广播、其他节点确认
   ↓
⑥  N 个确认块之后，tx 视为"最终 confirmed"（防重组）
💡 面试暴击点：tx hash ≠ tx 上链。拿到 hash 只代表进了 mempool。receipt 出现才代表打包完成（block hash 已存在）。很多新手把这两个概念搞混——面试问 「你怎么判断一笔交易成功了」时，答「等 receipt.status === 'success' 且有 N 个 confirmation」，而不是「拿到 tx hash」。

2.2 Gas 三件套（EIP-1559 现代模型）

2021 年伦敦硬分叉之前，用的是 legacy 模式：gasPrice 一个值走天下。用户自己出价，矿工挑高的打包，波动巨大。

EIP-1559 拆成 3 个值，让费用更可预测：

字段含义谁决定
gasLimit这笔 tx 最多能用多少 gas 单位你/钱包（估算）
baseFee本区块的基础 fee（每 block 根据拥堵动态调整），付给协议 销毁协议
maxPriorityFeePerGas小费，给矿工/验证者的激励，越高越优先你
maxFeePerGas你愿意付的每 gas 上限 = baseFee + priorityFee 的封顶你

实际支付：gasUsed × effectiveGasPrice，其中 effectiveGasPrice = min(maxFeePerGas, baseFee + maxPriorityFeePerGas)。你付的通常比预估少——因为 baseFee 可能比你的上限低。

面试题：「EIP-1559 之后，用户为什么不会像以前那样疯狂 overpay？」
答：baseFee 由协议按上一区块拥堵动态调整（每单位气量目标 15M gas），用户只需要设一个 maxFee 上限，实际扣的钱是 baseFee + 小费。而且 baseFee 是被销毁的（不给矿工），这也是 ETH 通缩的一部分。

2.3 nonce：地址的"发送计数器"

每个地址在链上有一个 nonce，从 0 开始递增。发第 1 笔 tx nonce=0、第 2 笔 nonce=1……严格按顺序执行。

  • 如果 nonce=5 的 tx 卡在 mempool 没打包，nonce=6 的 tx 就算发出去也会永远 pending——它必须等前面那笔先上链
  • 可以用 同 nonce + 更高 gas 发一笔"替换 tx"（钱包 UI 里的 "cancel" 或 "speed up" 就是这原理）
  • wagmi 会自动帮你管 nonce，除非做 batch 或高级操作，一般不用手动传

3. 三个核心 hook（本节主角）

Hook作用底层 RPC
useEstimateGas模拟 tx 得出 gasLimit 预估（不发交易）eth_estimateGas
useFeeData拉当前 baseFee + priority fee 建议值eth_feeHistory / eth_gasPrice
useSendTransaction让钱包签名 + 广播 tx，返回 hasheth_sendTransaction
useWaitForTransactionReceipt轮询直到 receipt 出现（tx 被打包）eth_getTransactionReceipt

4. 关键代码解剖

4.1 估算 gas（预览，不发交易）

const value = amountValid ? parseEther(amount) : undefined

const gasEstimate = useEstimateGas({
  to: toValid ? (to as Address) : undefined,
  value,
  query: { enabled: canPrepare },  // 输入没 ready 时别打 RPC
})

const feeData = useFeeData({ query: { enabled: isConnected } })

// 手续费上限 = gasLimit × maxFeePerGas
const estimatedFee =
  gasEstimate.data !== undefined && feeData.data?.maxFeePerGas
    ? gasEstimate.data * feeData.data.maxFeePerGas
    : undefined

bigint 运算：gasEstimate.data、maxFeePerGas 都是 bigint。直接 * 是 bigint 乘法，结果还是 bigint。不要 Number(x) 强转，超过 2^53 就精度丢失。

4.2 发送（触发钱包签名 + 广播）

const send = useSendTransaction()

<button onClick={() => {
  send.sendTransaction({
    to: to as Address,
    value: parseEther(amount),   // ETH 转 wei（乘 10^18）
  })
}}>
  {send.isPending ? '等 MM 签名…' : '发送交易'}
</button>

// send.data 是 tx hash（0x + 64 hex chars）
// send.error 是各种失败：用户拒签、余额不足、gas 不够…

4.3 等 receipt（tx 上链确认）

const receipt = useWaitForTransactionReceipt({
  hash: send.data,           // 传入 tx hash
  confirmations: 1,          // 等几个确认块（测试网 1 够了）
})

// 状态字段：
// receipt.isFetching     — 正在轮询等打包
// receipt.isSuccess      — receipt 拿到了
// receipt.data.status    — 'success' | 'reverted'
// receipt.data.blockNumber, gasUsed, effectiveGasPrice…
❓ 为什么不用 setInterval(() => eth_getTransactionReceipt, 3000) 手写轮询？

❌ 会撞免费 RPC 的限速（Alchemy 免费版 25 req/s，多开几个 tab 就废）
❌ tab 切走时不会自动 pause，浪费流量
❌ 组件卸载时容易忘 clear，内存泄漏
❌ 遇到临时网络错要自己写 retry / backoff

✅ useWaitForTransactionReceipt 内部：blockNumber tracking（新块出现才检查）+ retry with backoff + tab visibility 感知 + auto cleanup on unmount。轮子已经造好，别再造。

6. 面试要能答的 5 个问题

  1. 「tx hash 出现代表交易成功了吗？」
    不。hash 只代表签名+广播完成，进了 mempool。tx 是否成功要看 receipt：receipt.status === 'success'。而且看到 receipt 之后最好再等 N 个 confirmation 防重组。
  2. 「EIP-1559 之前和之后 gas 模型有什么区别？」
    之前：单一 gasPrice，用户拍脑袋出价，波动巨大，overpay 严重。
    之后：拆成 baseFee（协议决定、销毁）+ priority fee（给验证者的小费）。用户设 maxFee 上限，实际扣的钱是 baseFee + 小费。
  3. 「一笔 tx 卡在 mempool 怎么办？」
    用同一个 nonce 发一笔 gas 更高的 tx 替换掉它。转 0 ETH 给自己就是"cancel"；转原目标但提高 gas 就是"speed up"。钱包 UI 里的这两个按钮就是这原理。
  4. 「tx reverted 和 tx failed 有什么区别？」
    reverted：tx 上链了、gas 被扣了，但合约执行时抛异常。receipt.status = 'reverted'。
    failed（没上链）：签名错、余额不到 gas、nonce 冲突……这类根本进不了 block，钱包直接报错。
  5. 「为什么原生转账 gasLimit 是 21000？」
    EVM 每种操作都有 gas cost 表：一笔"简单转账"消耗固定的 21000 gas（包含签名验证、状态更新等基础操作）。合约调用会额外累加每条 opcode 的 gas。gasLimit 21000 是原生转账的硬底线。

7. 20% 原理，够撑面试第二层

  • parseEther / formatEther：parseEther('1') = 10n**18n。这是 wei（最小单位）。发交易 value 字段用 wei，读余额 balance.value 也是 wei，展示时才用 formatEther。永远用这两个函数，不要自己乘 10^18（可能精度丢失或写错幂）。
  • 为什么估算和实际 gas 会有差？useEstimateGas 用「当前状态」模拟，但 tx 真上链时状态已经变了（有人抢先）。所以钱包通常会自动给 gasLimit 加 buffer（+20%~50%），wagmi 也会。
  • 为啥用 useFeeData，为啥有 gasPrice / maxFeePerGas 两组值？兼容 legacy 网络（部分 L2/侧链没 EIP-1559）。主流 EVM 都上了 1559，直接用 maxFee + priorityFee 那组。
  • tx hash 是怎么算的？Keccak256(RLP(signed_tx))。是 tx 内容的哈希——同一笔 tx 在任何节点算出来都一样，所以能用作全网唯一 ID。
  • 为什么 confirmations 默认建议 > 1？防区块重组（reorg）。链偶尔会出现小分叉，短暂时刻 block N 是 A，几秒后变成 B，A 里的 tx 就"回滚"了。等 N 个确认之后 reorg 概率指数下降。主网大额交易一般 12+。
