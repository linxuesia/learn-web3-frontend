Lesson 01 · 第一支能跑的 DApp

连接钱包 + 读地址 + 读原生代币余额 · 目标 25 分钟内跑通

为什么先学这个？ 一个月要出能演示的 DApp，最贵的第一步是 你手上真的有一个跑得起来的项目骨架。所有后续（发交易、调合约、读事件）都长在这个骨架上。别在这节纠结原理，先让它跑。

1. 前置准备（一次性，5 分钟）

  • 已装 Node 20+（node -v 看一下）。
  • Chrome/Brave/Arc 已装 MetaMask。
  • MetaMask 里切到 Sepolia 测试网（Settings → Advanced → 打开 Show test networks）。
  • 拿点 Sepolia 测试币：Google Cloud Faucet 或 PoW Faucet。0.05 ETH 够用一个月。

1.5 如何确认当前连的是哪个网络？（必备小技能）

MetaMask 有时候点了切换网络 UI 没刷新，容易让人误判。这一段以后会反复用——永远不要靠 UI 感觉判断当前在哪个网上，用下面这几种方式一秒验证。

方法 A：看 MetaMask 里的三处标识（最快）

  • 顶部网络下拉按钮：会写着 Sepolia / Ethereum Mainnet 等。
  • 余额那行的代币符号：主网显示 ETH，Sepolia 显示 SepoliaETH，Base Sepolia 显示 ETH（但网络名不同）。这行比顶部按钮更不容易骗人。
  • 底部胶囊按钮：同样写着当前网络名。

方法 B：Console 一行代码验（最硬核，面试也可能问）

打开任意正常网页（比如 sepolia.etherscan.io 或 ethereum.org——不能是 chrome:// 开头的内部页，MetaMask 不往那些页面注入），F12 → Console → 粘：

await window.ethereum.request({ method: 'eth_chainId' })

拿到十六进制的 chainId 后，对照下面这张表。或者一次查全套：

const chainId = await window.ethereum.request({ method: 'eth_chainId' })
const accounts = await window.ethereum.request({ method: 'eth_accounts' })
console.log({ chainId, chainIdDec: parseInt(chainId, 16), accounts })
坑点：如果报 Cannot read properties of undefined (reading 'request')，说明这个页面没被注入 window.ethereum——不是钱包坏了，是你在错的页面按 F12 了（新标签页、chrome://extensions、about:blank 都不会被注入）。换个普通网站再试。

方法 C：chainlist.org 直连测试

打开 chainlist.org（勾上 Testnets）→ 点 "Connect Wallet"，它直接把当前网络亮出来，还能一键给 MetaMask 加新链。

3. 配置 wagmi（5 分钟）

建 src/wagmi.ts：

import { http, createConfig } from 'wagmi'
import { sepolia } from 'wagmi/chains'
import { injected } from 'wagmi/connectors'

export const config = createConfig({
  chains: [sepolia],
  connectors: [injected()],
  transports: {
    [sepolia.id]: http(),
  },
})

改 src/main.tsx，把 App 包在 wagmi + react-query 的 Provider 里：

import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import { WagmiProvider } from 'wagmi'
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'
import { config } from './wagmi'
import App from './App'
import './index.css'

const queryClient = new QueryClient()

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <WagmiProvider config={config}>
      <QueryClientProvider client={queryClient}>
        <App />
      </QueryClientProvider>
    </WagmiProvider>
  </StrictMode>,
)

4. 写「连钱包 + 读余额」的 UI（8 分钟）

把 src/App.tsx 换成：

import {
  useAccount, useConnect, useDisconnect, useBalance, useChainId,
} from 'wagmi'
import { formatEther } from 'viem'

export default function App() {
  const { address, isConnected } = useAccount()
  const chainId = useChainId()
  const { connectors, connect, status, error } = useConnect()
  const { disconnect } = useDisconnect()
  const { data: balance } = useBalance({ address })

  return (
    <main style={{ maxWidth: 640, margin: '3rem auto', fontFamily: 'system-ui' }}>
      <h1>Hello Web3</h1>

      {!isConnected ? (
        <div>
          {connectors.map((c) => (
            <button key={c.uid} onClick={() => connect({ connector: c })}>
              连接 {c.name}
            </button>
          ))}
          {status === 'pending' && <p>连接中…</p>}
          {error && <p style={{ color: 'crimson' }}>{error.message}</p>}
        </div>
      ) : (
        <div>
          <p>地址：<code>{address}</code></p>
          <p>当前链 ID：{chainId}</p>
          <p>
            余额：
            {balance
              ? \`\${formatEther(balance.value)} \${balance.symbol}\`
              : '读取中…'}
          </p>
          <button onClick={() => disconnect()}>断开</button>
        </div>
      )}
    </main>
  )
}

  • 点「连接 Injected / MetaMask」→ MetaMask 弹窗 → 授权。
  • 页面上应出现你的地址、Chain ID = 11155111（Sepolia）、Sepolia ETH 余额。
  • 断开后按钮变回「连接」。

如果这几步都成立，恭喜——你已经跑通了 web3 前端的最短闭环：钱包 → 前端 → 链 → 数据 → UI。所有后续内容都是在这条链路上加东西。

6. 20% 的原理，够撑面试第一层

  • 钱包为什么能被网页读到？ 因为浏览器钱包会往页面注入一个符合 EIP-1193 的 provider（早期是 window.ethereum，现代是 EIP-6963 多钱包发现）。wagmi 的 injected() 就是在这层之上做的封装。
  • viem vs ethers vs wagmi？ viem 是底层 EVM 客户端；wagmi 是它的 React hooks 封装；ethers 是老前辈，很多老项目还在用，但新项目 2024 起主流是 wagmi + viem。
  • 为什么要 react-query？ wagmi 的 hooks 底层用 react-query 做缓存和 refetch，链上数据本质是"网络请求"，缓存/失效逻辑复用 react-query 是最省心的方案。

🚨 真实踩坑记录 · MetaMask 的 per-site network（超重要，面试必答）

症状：页面上明明连了钱包，去 MetaMask UI 里手动切换网络后，DApp 页面上的 chainId 纹丝不动。Console 里连 window.ethereum.on('chainChanged', ...) 都收不到事件。你以为是钱包卡了、代码错了、wagmi 挂了——都不是。

Root Cause（第一性原理）

先回到基本定义：

  • window.ethereum 是钱包注入到某个 origin（比如 http://localhost:5173）的 provider 对象，它不是"钱包本身"，是"钱包派给这个页面的一个代表"。
  • EIP-1193 里 eth_chainId 的定义原文："the chain ID of the chain currently used by the wallet for this connection"。
  • chainChanged 事件的定义："emitted when the chain has changed for this site"。

关键词全在 "for this site"。

MetaMask 从 v12 (2024) 开始实现了 Network per Site——每个 dApp 独立记录自己"当前"用哪个网络。你在 MM 主 UI 顶部切网络，切的是钱包主界面显示的默认网络，不是 localhost:5173 这个 site 当前的网络。所以：

  • MM UI 上看着切了 ✅
  • 但对 localhost:5173 这个 origin 来说，链并没有变；
  • 所以 eth_chainId 返回值不变；
  • 所以 chainChanged 事件本来就不该 emit（链根本没变），DApp 收不到很正常。

这不是 bug，是 MetaMask 有意的隔离设计——避免你在一个 DApp 上操作时，另一个 tab 里忘了切链导致误签名/误转账。

正确姿势：DApp 主动请求切链

EIP-1193 + EIP-3326（wallet_switchEthereumChain）+ EIP-3085（wallet_addEthereumChain）定义的标准做法：

// 让钱包为当前 site 切到主网（如果 MM 里没这个网络，会先弹加链弹窗）
await window.ethereum.request({
  method: 'wallet_switchEthereumChain',
  params: [{ chainId: '0x1' }]   // 十六进制，不是十进制
})

用户在 MetaMask 弹窗里点确认——这一刻链对 localhost:5173 才真的变了，才会 emit chainChanged，wagmi 才会收到、UI 才会更新。

wagmi 里怎么写

import { useSwitchChain, useChainId } from 'wagmi'
import { mainnet, sepolia } from 'wagmi/chains'

function ChainSwitcher() {
  const chainId = useChainId()
  const { chains, switchChain, isPending, error } = useSwitchChain()

  return (
    <>
      {chains.map(c => (
        <button
          key={c.id}
          disabled={isPending || c.id === chainId}
          onClick={() => switchChain({ chainId: c.id })}
        >
          {c.id === chainId ? '✓ ' : ''}{c.name}
        </button>
      ))}
      {isPending && <p>等 MM 弹窗…</p>}
      {error && <p style={{color:'crimson'}}>{error.message}</p>}
    </>
  )
}

switchChain 底层就是调 wallet_switchEthereumChain，wagmi 只是帮你包一层 React state。

Uniswap / OpenSea / Aave 都怎么做的？

去看它们的 UI——永远有一个"切链"下拉菜单在钱包按钮旁边，不会指望用户去 MM 里手动切。这是行业标准做法：

  • DApp 自己知道当前应该在哪个链（比如 Base Sepolia）
  • 用户连钱包时如果不在这个链，UI 提示"请切到 Base Sepolia" + 一键切链按钮
  • 用户点了 → 调 switchChain → MM 弹窗 → 用户确认 → 链切换 → UI 更新

常见问题 & 排错顺序

症状排查
调 switchChain 报 Unrecognized chain ID目标链没在 MM 里，先调 wallet_addEthereumChain；wagmi 里 chains 数组包含这个链，一般会自动加
调用后一直 pending 没弹窗MM 扩展 service worker 睡了，点 MM 图标唤醒；或 F5 刷新页面
用户点了"取消"会抛 UserRejectedRequestError（code 4001），要 catch 并友好提示
DApp 只支持特定链，用户切到 unsupported用 chainId in supported 判断，显示"请切到 XX 链"横幅 + 引导按钮

💡 面试可能问到

  • Q: DApp 里怎么让用户切链？
    A: 调 wallet_switchEthereumChain（EIP-3326），钱包会弹窗让用户为当前 origin 确认。链不存在的话先调 wallet_addEthereumChain（EIP-3085）。wagmi 封装成 useSwitchChain。
  • Q: 用户在 MetaMask UI 里手动切网络，DApp 收得到 chainChanged 吗？
    A: MetaMask v12+ 有 per-site network 隔离，用户在 MM UI 顶部切的是钱包默认网络，不是当前 dApp 的网络。所以 dApp 的 eth_chainId 不变，chainChanged 也不会 emit。这不是 bug，是安全隔离。
  • Q: wallet_switchEthereumChain 和 chainChanged 事件的关系？
    A: 前者是 dApp → 钱包的请求，后者是钱包 → dApp 的通知。成功切链会依次触发：dApp 调 wallet_switchEthereumChain → 钱包弹窗 → 用户确认 → 钱包 emit chainChanged(newChainId) → 所有订阅者收到 → dApp 状态更新。
sia 的踩坑时间线（真实）：
1. 写好代码，MM 里手动切网，UI 不动 →
2. 加了 window.ethereum.on('chainChanged', ...)，切网仍不打印 →
3. 用 eth_chainId 直查，MM 里切网后返回值不变 →
4. 意识到：链对这个 site根本没变，事件当然不发 →
5. 改用 wallet_switchEthereumChain，弹窗一路点确认，chainChanged 立刻打出来，UI 秒变。
这段的价值：真实工程里 90% 的"钱包 bug"都是这类"设计 vs 直觉"的错位。

🔎 深挖 · 为什么"Injected"按钮出现快，"MetaMask"按钮慢一点（EIP-1193 vs EIP-6963）

现象：Lesson 01 页面上如果同时出现两个连接按钮，你会发现 Injected 按钮几乎瞬间就有，MetaMask 按钮要晚 200-300ms 才出现或才能点。这是钱包发现机制的架构升级带来的必然代价。

EIP-1193 时代（老，2018）：window.ethereum 单一坑位

最早的钱包注入协议只规定一件事：钱包把自己塞到 window.ethereum。

window.ethereum = <MetaMask provider>
await window.ethereum.request({ method: 'eth_requestAccounts' })

问题很明显：只有一个坑位。你装了 MetaMask + Coinbase + OKX + Rabby，谁后注入谁覆盖前面的。用户想连哪个只能 disable 其他扩展——这就是历史上臭名昭著的"钱包大战"（wallet war）。

有些钱包搞骚操作，把 window.ethereum 改成 window.ethereum.providers 数组，各家实现不统一，一坨屎。

wagmi 的 injected() connector 就是给这套老协议兜底的：同步读 window.ethereum——0 延迟，所以你觉得它快。缺点：连的是当前抢到 window.ethereum 那把钱包，可能不是用户想连的那个。

EIP-6963 时代（新，2023）：双向事件握手

EIP-6963（Multi Injected Provider Discovery）标准化了一套广播-通告协议，让多个钱包共存不打架：

  1. 钱包自曝：页面加载时，每个钱包主动 dispatchEvent('eip6963:announceProvider')，带上 {uuid, name, icon, rdns, provider} 元数据。
  2. DApp 反向询问：补漏——DApp 也发一次 eip6963:requestProvider，让"慢启动"的钱包再 announce 一次。
  3. 钱包再次响应：收到 requestProvider → 再 announce 一次。
  4. DApp 收集：监听 announceProvider 累积 200-300ms → 拿到完整钱包列表。

wagmi 的 metaMask() connector 走的就是这套。那 200-300ms 就是"discovery window"，你观察到的延迟就来源于此。

Lesson 01 里怎么改（已经改了）

看 src/wagmi.ts：

import { injected, metaMask } from 'wagmi/connectors'

connectors: [
  metaMask(),   // EIP-6963 + MetaMask SDK · 主推
  injected(),   // EIP-1193 · 兜底老钱包
]

UI 里 useConnect().connectors 会同时返回这两个 connector（wagmi 内部会去重，如果 EIP-6963 已识别到 MM，同源的 injected 有时会被合并）。

生产环境的三种优化

  • 骨架屏：连接按钮区显示 loading skeleton，等 EIP-6963 discovery 一收齐再渲染。用户感受"就是在扫描钱包"而不是"卡了"。
  • 用 RainbowKit / ConnectKit：这些库把 EIP-6963 收集、UI 骨架、图标资源、错误处理全封装好——生产 DApp 90% 用这两个之一。
  • 双兜底：就是我们现在的写法 [metaMask(), injected()]——新协议主推、老协议兜底。

💡 面试题（这次踩坑直接三道）

  • Q: EIP-6963 解决什么问题？
    A: EIP-1193 时代 window.ethereum 单一坑位，多钱包互相覆盖。EIP-6963 用双向事件广播机制，每个钱包独立 announce 自己（带 uuid/rdns/name/icon 元数据），DApp 通过监听事件收集所有可用 provider。
  • Q: EIP-6963 有什么代价？
    A: 有 discovery window 延迟（~200-300ms），DApp 要等钱包 announce 到齐。主流做法先渲染 skeleton，收齐再显示可选列表。老 injected 快但不安全。
  • Q: 2026 年做 DApp 应该选哪个协议？
    A: EIP-6963 主推（走 metaMask() / RainbowKit / ConnectKit），EIP-1193 injected 做兜底。裸调 window.ethereum.request 已经过时。
sia 的观察时间线：
"我发现页面上有两个按钮，Injected 出现快，MetaMask 慢一点"——这个观察 100% 命中了两代协议的架构差异。EIP-1193 是同步读全局变量的 O(1) 操作，EIP-6963 是异步事件握手，多了几个 ms 但换来多钱包共存和元数据。
